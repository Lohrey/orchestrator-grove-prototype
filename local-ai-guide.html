Installed local layout

Use a shared runtime venv per Torch/CUDA stack, then point tools at the matching runtime.

G:\Coding\Projects\local inference\
  runtimes\
    torch290-cu128-py311\
  tabby-exllama\
  models\
    Qwen2.5-Coder-7B-Instruct-exl2-6_5\

Do not reuse Ollama model files in ExLlama. Ollama commonly uses GGUF-style artifacts. ExLlama expects EXL2, EXL3, or safetensors-compatible model folders.

Prompt layout for cache reuse

Put stable prompt text first. Prefix cache reuse is strongest when the beginning of the prompt is byte-for-byte identical between calls.

  1. System rules
  2. Stable JSON contract
  3. Stable DSL grammar and global action rules
  4. Stable action format conventions
  5. Selected action docs for this turn
  6. Current game state and variables
  7. Recent conversation
  8. Latest player input

Context and caching notes

  • 12k tokens is roughly 8k-9.5k English words, and less for code/JSON/DSL-heavy text.
  • Cache reuse mainly speeds prompt processing, not token-by-token answer generation.
  • For this game, keep invariant grammar/schema static and append smart-selected action docs after it.
  • Multiple browser requests may arrive at once, but one consumer GPU is still a scarce generation resource. Queue NPC calls intentionally.

Ollama vs TabbyAPI / ExLlama

Ollama is easiest for users and GGUF models. TabbyAPI with ExLlama is faster on NVIDIA GPUs when using EXL2/EXL3 models, but setup is more specific.

For strict DSL-to-JSON, the model matters as much as the backend. Qwen coder models are a better first target than modified Gemma variants.

NVIDIA ACE SLMs

ACE/Nemotron small models are promising for low-latency NPC dialogue and roleplay, but they are not the first recommendation for large DSL docs plus strict JSON action output.

Use them experimentally as a second-stage flavour/dialogue model after a coder model has produced validated action JSON.
